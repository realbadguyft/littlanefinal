# FRESHERS TAKEOVER — Ticket Booking Setup

This project now has a full "Book Ticket" flow: pay → get a PDF pass emailed
+ downloadable → every sale logged in an admin dashboard.

## 1. Install dependencies

```bash
# frontend deps (if any)
npm install

# backend deps
cd server
npm install
```

## 2. Run it

Your Razorpay **test** credentials are already set in `server/.env`, so the
server starts in **LIVE mode** (real Razorpay checkout modal, test cards
only — no real money moves). If you ever remove/rename `.env`, it falls back
to Test Mode (simulated instant payment success) so you can still try the
flow without any gateway at all.

```bash
cd server
npm start
```

Then open `index.html` in your browser (or serve the root folder with any
static server) and click the ticket button. `script.js` is already pointed
at `http://localhost:3000` via `window.LITTLANE_API_BASE` in `index.html` —
change that if you host the API elsewhere.

## 3. Add real email delivery

Razorpay is already configured. To actually email tickets (instead of the
temporary preview inbox), edit `server/.env`:

- **SMTP_HOST / SMTP_PORT / SMTP_USER / SMTP_PASS / EMAIL_FROM** — your email
  provider's SMTP details (a Gmail "App Password" works well)
- **ADMIN_KEY** — change this to your own private password for `/admin`
- **BASE_URL** — the public URL your server runs on in production (used for
  ticket download links)

Restart the server after editing `.env`. When you're ready for real payments,
swap `RAZORPAY_KEY_ID`/`RAZORPAY_KEY_SECRET` for your **live** keys from the
Razorpay dashboard (the test keys currently in `.env` only work with test
cards — no real money moves).

## 4. Monitor every sale

Go to `http://localhost:3000/admin` (or your deployed URL + `/admin`) and
enter your `ADMIN_KEY`. You'll see, live:

- Every order: name, email, phone, gender, quantity, amount, timestamps
- Payment ID and Ticket ID for each sale
- Whether the ticket email was sent, and the exact error if it failed
- Total revenue, paid orders, email failures, ticket-generation failures
- A "Resend" button to retry a failed ticket email

To change prices, edit the `PRICING` object near the top of `server/server.js`
and the two `<option>` labels + price line in the booking popup in
`index.html`.

## How it works, end to end

1. Buyer fills the form → `POST /api/create-order` (server computes the real
   price server-side, never trusts the browser).
2. Razorpay checkout opens (or is skipped in Test Mode).
3. On payment success → `POST /api/verify-payment`:
   - Signature is verified (live mode) so payments can't be faked.
   - A unique Ticket ID + QR code + PDF pass are generated.
   - The PDF is emailed to the buyer; the API response also returns a direct
     download link regardless of whether the email succeeded.
   - Every step (paid, ticket generated, email sent/failed) is logged to
     `server/data/sales.json` with a timestamp and, on failure, the exact
     error message.
4. Admin dashboard (`/admin`) reads that same log in real time.

## Ticket design

The ticket (PDF, email, and in-app success screen) is styled after the
"Littix" reference design: banner artwork, NOT SCANNED / ACTIVE badges,
ticket type/qty/price row, ticket ID, QR code, and the Littix wordmark.

- Banner artwork lives at `images/freshers-takeover-banner.png` (frontend)
  and `server/ticket-banner.png` (used in the PDF/email — keep these two in
  sync if you swap the artwork).
- Event details (date, time, venue, "Generated by" name) are placeholders
  (`TBA`) — edit the `EVENT_DETAILS` object near the top of
  `server/ticket.js` with your real date/time/venue.
